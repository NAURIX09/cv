Introduction
Qu'est-ce que CVMatch IA ?

CVMatch IA est une plateforme web complète qui utilise l'Intelligence Artificielle pour mettre en relation des candidats et des recruteurs. Elle analyse les CV automatiquement et calcule des scores de compatibilité.

🏗️ L'architecture en deux couches

Le projet est divisé en deux applications distinctes qui communiquent ensemble :

🌐
Couche Interface (Frontend)
HTML + CSS + JavaScript — Ce que l'utilisateur voit dans son navigateur
↕
🐘
Couche Web (PHP)
PHP — Gère les pages, les formulaires, la connexion, les sessions. Tourne sur Apache/Nginx.
↕ HTTP (cURL)
🐍
Couche API (Python / FastAPI)
Python — Analyse les CV, calcule les scores, gère l'IA. Tourne sur le port 8000.
↕ SQL
🗄️
Couche Données (MySQL)
MySQL — Stocke tout : candidats, recruteurs, CV, messages, historique des conversations.
↕ API
🤖
Couche IA Externe (Groq)
Groq API — LLM Llama 3 pour l'analyse, résumés, conversations. Sentence-Transformers pour les embeddings.
💡 Analogie simple

Imagine un restaurant. PHP = le serveur qui prend les commandes et sert les plats. Python = le cuisinier qui prépare les plats complexes. MySQL = le réfrigérateur qui stocke tout. Groq/IA = le chef étoilé consulté pour les recettes sophistiquées.

Fichier PHP
config.php — Le cerveau central

Ce fichier est inclus dans tous les autres fichiers PHP. C'est le point de départ de tout. Il définit les paramètres de connexion, les fonctions partagées, et les outils utilisés partout.

🎯 Principe clé : require_once

Chaque fichier PHP commence par require_once 'config.php'. Ça signifie "inclure ce fichier une seule fois". Ainsi toutes les fonctions et constantes définies ici sont disponibles partout dans l'application.

1. Les constantes define()

Les constantes sont des valeurs fixes qui ne changent jamais pendant l'exécution. On les définit une fois, on les utilise partout.

PHPconfig.php — Constantes de configuration
// ── Base de données ──────────────────
define('DB_HOST',    'localhost');   // Adresse du serveur MySQL
define('DB_NAME',    'cvmatch_ia'); // Nom de la base de données
define('DB_USER',    'root');       // Utilisateur MySQL
define('DB_PASS',    '');          // Mot de passe (vide en développement)
define('DB_CHARSET', 'utf8mb4');   // Encodage : supporte emojis, accents, etc.

// ── API Python (FastAPI) ──────────────
define('PYTHON_API', 'http://localhost:8000'); // Adresse de l'API Python
define('API_KEY',    'cvmatch-secret-2025');  // Clé secrète partagée PHP↔Python

// ── Upload ────────────────────────────
define('UPLOAD_DIR',   __DIR__ . '/uploads/');  // __DIR__ = dossier du fichier actuel
define('MAX_FILE_SIZE', 5 * 1024 * 1024);        // 5 Mo en bytes (5 × 1024 × 1024)
define('ALLOWED_EXT',  ['pdf','docx','jpg','jpeg','png']); // Extensions autorisées
__DIR__ . '/uploads/'
__DIR__ est une "constante magique" PHP. Elle contient automatiquement le chemin absolu du dossier où se trouve ce fichier. Exemple : /var/www/cvmatch/public. On y ajoute /uploads/ pour obtenir le chemin complet du dossier d'upload.
5 * 1024 * 1024
Calcul de 5 mégaoctets en bytes. 1 Ko = 1024 bytes. 1 Mo = 1024 Ko = 1024×1024 bytes. 5 Mo = 5×1024×1024 = 5 242 880 bytes. PHP doit comparer la taille du fichier avec ce nombre.

2. La fonction db() — Connexion PDO Singleton

PDO (PHP Data Objects) est la façon moderne et sécurisée de parler à MySQL en PHP. Le pattern "Singleton" garantit qu'on ne crée qu'une seule connexion, réutilisée à chaque appel.

PHPFonction db() — Connexion unique à MySQL
function db(): PDO {
    static $pdo = null;     // "static" = mémorisé entre les appels
    if ($pdo === null) {    // Si pas encore connecté...
        try {
            $pdo = new PDO(
                'mysql:host='.DB_HOST.';dbname='.DB_NAME.';charset='.DB_CHARSET,
                DB_USER, DB_PASS,
                [
                    PDO::ATTR_ERRMODE            => PDO::ERRMODE_EXCEPTION, // Lance une exception si erreur SQL
                    PDO::ATTR_DEFAULT_FETCH_MODE => PDO::FETCH_ASSOC,       // Résultats en tableau associatif
                    PDO::ATTR_EMULATE_PREPARES   => false,                 // Requêtes préparées réelles (sécurité)
                ]
            );
        } catch (PDOException $e) {
            die('<div>Erreur MySQL : '.$e->getMessage().'</div>'); // Arrête tout et affiche l'erreur
        }
    }
    return $pdo;   // Retourne la connexion (neuve ou existante)
}
💡 C'est quoi "static" ?

Normalement, une variable dans une fonction est détruite quand la fonction se termine. Avec static, la variable persiste entre les appels. Ainsi $pdo est créé une fois et réutilisé, évitant de re-connecter à MySQL à chaque fois.

💡 FETCH_ASSOC c'est quoi ?

Quand MySQL retourne des données, on peut les recevoir de deux façons. FETCH_ASSOC donne un tableau avec les noms de colonnes : ['nom' => 'Kouamé', 'email' => '...']. C'est plus lisible que des index numériques.

3. Les fonctions de session et sécurité

PHPGestion des sessions — Sécurité
function startSession(): void {
    if (session_status() === PHP_SESSION_NONE) {  // Si pas encore démarrée
        session_start([
            'cookie_httponly' => true,    // Cookie inaccessible via JavaScript → protection XSS
            'cookie_samesite' => 'Strict', // Cookie envoyé seulement depuis notre site → protection CSRF
        ]);
    }
}

function requireLogin(string $role = ''): void {
    startSession();
    if (empty($_SESSION['user_id'])) {              // Pas connecté ?
        header('Location: index.php'); exit;        // Redirection vers l'accueil
    }
    if ($role && ($_SESSION['role'] ?? '') !== $role) { // Mauvais rôle ?
        header('Location: index.php'); exit;        // Redirection aussi
    }
}
🔐 Sécurité importante

La fonction requireLogin('recruteur') en haut du dashboard_recruteur.php garantit qu'un candidat ne peut jamais accéder à l'espace recruteur même s'il tape l'URL directement dans son navigateur. C'est une vérification côté serveur, impossible à contourner côté navigateur.

4. La fonction callAPI() — Communication PHP→Python

PHPcURL — Appel de l'API Python depuis PHP
function callAPI(string $route, array $data = [], string $method = 'POST'): ?array {
    $url = PYTHON_API . $route;          // Ex: http://localhost:8000/candidats/upload/42
    $ch  = curl_init($url);             // Initialise une requête HTTP
    curl_setopt_array($ch, [
        CURLOPT_RETURNTRANSFER => true,  // Retourner la réponse (pas l'afficher)
        CURLOPT_POST           => true,  // Méthode POST
        CURLOPT_POSTFIELDS     => json_encode($data),  // Corps de la requête en JSON
        CURLOPT_HTTPHEADER     => [
            'Content-Type: application/json',     // On envoie du JSON
            'X-API-Key: ' . API_KEY,              // Clé secrète d'authentification
        ],
        CURLOPT_TIMEOUT        => 60,     // Abandon si pas de réponse en 60 secondes
    ]);
    $resp = curl_exec($ch);             // Envoie la requête et attend la réponse
    $code = curl_getinfo($ch, CURLINFO_HTTP_CODE); // Récupère le code HTTP (200, 404...)
    curl_close($ch);                    // Ferme la connexion
    if ($code !== 200 || !$resp) return null; // Erreur → retourner null
    return json_decode($resp, true);  // Convertit la réponse JSON en tableau PHP
}
💡 cURL, c'est quoi ?

cURL est une bibliothèque qui permet à PHP de faire des requêtes HTTP, exactement comme un navigateur. Quand tu vas sur google.com, ton navigateur fait une requête GET. Ici, PHP fait une requête POST vers l'API Python — même principe, mais depuis le code serveur.

5. La fonction uploadCV() — Sécurisation des uploads

PHPUpload sécurisé de fichiers
function uploadCV(array $file, int $userId): array {
    $ext = strtolower(pathinfo($file['name'], PATHINFO_EXTENSION)); // Extrait 'pdf' de 'monCV.PDF'
    if (!in_array($ext, ALLOWED_EXT)) {
        return ['ok'=>false, 'error'=>'Format non autorisé'];     // Rejette les fichiers non-autorisés
    }
    if ($file['size'] > MAX_FILE_SIZE) {
        return ['ok'=>false, 'error'=>'Fichier trop volumineux'];   // Rejette les fichiers > 5 Mo
    }
    // Génère un nom unique pour éviter les collisions
    $nom = 'cv_' . $userId . '_' . time() . '.' . $ext; // Ex: cv_42_1718000000.pdf
    $dest = UPLOAD_DIR . $nom;
    if (!move_uploaded_file($file['tmp_name'], $dest)) { // Déplace depuis le dossier temp
        return ['ok'=>false, 'error'=>'Erreur enregistrement'];
    }
    return ['ok'=>true, 'nom'=>$nom, 'ext'=>$ext];          // Succès
}
🛡️ Pourquoi renommer le fichier ?

Un utilisateur malveillant pourrait uploader un fichier nommé hacker.php. Si on garde le nom original, l'attaquant pourrait accéder à monsite.com/uploads/hacker.php et exécuter du code dangereux. En renommant avec cv_{userId}_{time}.ext, on élimine ce risque.

🔗 Les relations entre tables (FOREIGN KEY)

SQLClé étrangère — Lien entre cv_files et candidats
FOREIGN KEY (candidat_id) REFERENCES candidats(id) ON DELETE CASCADE
💡 ON DELETE CASCADE — c'est quoi ?

Si on supprime un candidat de la table candidats, tous ses CV dans la table cv_files sont automatiquement supprimés aussi. Sans ça, on aurait des orphelins : des CV qui référencent un candidat inexistant, ce qui créerait des bugs.

👁️ La vue SQL vue_candidats_complets

Une "vue" est comme une requête enregistrée sous un nom. Elle simplifie les requêtes complexes.

SQLVue — Jointure candidats + leur dernier CV
CREATE OR REPLACE VIEW vue_candidats_complets AS
SELECT
  c.id, c.nom, c.prenom, c.email, c.ville, c.annees_exp,
  cv.id       AS cv_id,
  cv.nom_fichier,
  cv.texte_brut,
  cv.embedding_json    -- Le vecteur IA du CV
FROM candidats c
LEFT JOIN cv_files cv ON cv.candidat_id = c.id
  AND cv.id = (
    SELECT MAX(cv2.id) FROM cv_files cv2 WHERE cv2.candidat_id = c.id
  );  -- Sous-requête : récupère seulement le DERNIER CV de chaque candidat
💡 LEFT JOIN vs JOIN

JOIN (ou INNER JOIN) retourne seulement les candidats qui ont un CV. LEFT JOIN retourne tous les candidats, même ceux sans CV (leur colonne cv_id sera NULL). On choisit LEFT JOIN pour voir tous les inscrits, y compris ceux qui n'ont pas encore uploadé de CV.

🔐 Pourquoi les mots de passe ne sont pas stockés en clair ?

PHPHachage et vérification du mot de passe
// Lors de l'inscription :
$hash = password_hash($pass, PASSWORD_BCRYPT);
// $hash = "$2y$10$ABCDEFGHIJK..." (chaîne illisible, irréversible)
// Stocké en base. JAMAIS le mot de passe original.

// Lors de la connexion :
if (password_verify($pass, $u['password_hash'])) {
    // Connexion autorisée — PHP compare sans jamais déchiffrer
}
🛡️ Sécurité fondamentale

BCrypt est un algorithme de hachage à sens unique. On ne peut pas retrouver le mot de passe à partir du hash. Même si un hacker vole la base de données, il ne peut pas lire les mots de passe. password_verify() refait le calcul et compare les résultats.

PHP
login_candidat.php & login_recruteur.php

Ces fichiers gèrent l'inscription et la connexion des utilisateurs. Ils utilisent le même principe : traitement du formulaire en PHP côté serveur, puis affichage HTML côté client.

🔄 Principe du formulaire PHP (POST/GET)

1
L'utilisateur visite la page (GET)
Le navigateur envoie une requête GET. PHP affiche le formulaire HTML vide. $_SERVER['REQUEST_METHOD'] === 'GET' à ce moment.
2
L'utilisateur soumet le formulaire (POST)
Le navigateur envoie une requête POST avec les données du formulaire. $_SERVER['REQUEST_METHOD'] === 'POST'. PHP entre dans le bloc if de traitement.
3
PHP valide les données
Vérifie que l'email est valide, que les champs sont remplis, que le mot de passe est assez long. En cas d'erreur, il stocke le message dans $error.
4
PHP crée/vérifie le compte et redirige
Si tout est ok : insère en BDD (inscription) ou vérifie le hash (connexion). Puis stocke l'identité dans $_SESSION et redirige vers le dashboard.
PHPlogin_candidat.php — Logique d'inscription complète
if ($_SERVER['REQUEST_METHOD']==='POST') {
    $action = $_POST['action'] ?? '';  // 'register' ou 'login' selon le bouton cliqué

    if ($action==='register') {
        // Lecture et nettoyage des données
        $email  = trim($_POST['email'] ?? '');    // trim() enlève les espaces avant/après
        $pass   = $_POST['password']   ?? '';
        $pass2  = $_POST['password2']  ?? '';

        // Validations en cascade
        if (!filter_var($email, FILTER_VALIDATE_EMAIL)) { // Vérifie format email
            $error = 'Email invalide.';
        } elseif (strlen($pass) < 6) {   // Minimum 6 caractères
            $error = 'Mot de passe trop court.';
        } elseif ($pass !== $pass2) {    // Les deux mots de passe doivent correspondre
            $error = 'Mots de passe différents.';
        } else {
            // Vérifier si l'email est déjà utilisé (UNIQUE en BDD)
            $check = db()->prepare("SELECT id FROM candidats WHERE email=? LIMIT 1");
            $check->execute([$email]);    // ? remplacé par $email — protection injection SQL
            if ($check->fetch()) {
                $error = 'Email déjà utilisé.';
            } else {
                $hash = password_hash($pass, PASSWORD_BCRYPT); // Hachage sécurisé
                $stmt = db()->prepare("INSERT INTO candidats (nom,prenom,email,password_hash) VALUES(?,?,?,?)");
                $stmt->execute([$nom, $prenom, $email, $hash]);  // Insère en base
                $id = (int)db()->lastInsertId();    // Récupère l'ID du nouveau candidat
                $_SESSION['user_id'] = $id;         // Mémorise l'ID en session
                $_SESSION['role']    = 'candidat';  // Mémorise le rôle
                header('Location: dashboard_candidat.php'); exit; // Redirection
            }
        }
    }
}
📚 Qu'est-ce qu'une SESSION PHP ?

Une session PHP est comme un badge d'accès temporaire. Quand tu te connectes, PHP crée un identifiant unique (ex: abc123xyz) stocké dans un cookie dans ton navigateur. Sur le serveur, PHP retient que abc123xyz correspond à l'utilisateur ID=42, rôle=candidat. À chaque page visitée, PHP vérifie ce badge. $_SESSION['user_id'] est la façon d'accéder à ces informations.

🔍 Dashboard Recruteur — Le cœur du système

La fonction runSearch()

JavaScriptRecherche IA — Communication avec l'API Python
async function runSearch() {
    const q = document.getElementById('searchInput').value.trim();
    if (!q) { showToast('Saisissez une requête', 'error'); return; }

    // Envoie la requête DIRECTEMENT à l'API Python (pas via PHP!)
    const resp = await fetch(API_BASE + '/recherche', {  // API_BASE = 'http://localhost:8000'
        method: 'POST',
        headers: {
            'Content-Type': 'application/json',
            'X-API-Key': API_KEY   // Clé secrète pour authentifier la requête
        },
        body: JSON.stringify({ texte_requete: q })  // Convertit objet JS → chaîne JSON
    });

    const data = await resp.json();   // Convertit la réponse JSON → objet JS
    currentResults = data.profils || [];  // Stocke les résultats dans une variable globale

    renderResults(currentResults, q); // Affiche les cartes candidats
}
⚡ Async/Await — C'est quoi ?

En JavaScript, les appels réseau prennent du temps (quelques secondes). Sans async/await, le code continuerait à s'exécuter pendant l'attente et essaierait d'afficher des résultats inexistants. await dit "attends que la réponse arrive avant de continuer". async devant la fonction indique qu'elle contient des await.

📚 try/finally — Pourquoi ?

Si une exception (erreur) se produit dans le bloc try, le bloc finally s'exécute quand même. Ainsi conn.close() est toujours appelé, même en cas d'erreur. Sans ça, les connexions pourraient rester ouvertes indéfiniment et épuiser le pool.

Python_extraire_json() — Nettoyage de la réponse IA
def _extraire_json(contenu: str) -> dict:
    # L'IA peut envelopper le JSON dans des blocs ```json...``` — on enlève ça
    contenu = re.sub(r'```(?:json)?\s*', '', contenu.strip())
    contenu = re.sub(r'```\s*', '', contenu)

    # Trouve le premier {…} dans la réponse (regex)
    m = re.search(r'\{.*\}', contenu, re.DOTALL)  # DOTALL = le . matche aussi les retours à la ligne
    if m:
        try: return json.loads(m.group())  # Convertit JSON → dictionnaire Python
        except: return {}
    return {}  # Retourne dictionnaire vide si pas de JSON trouvé
💡 Temperature en IA — c'est quoi ?

La "temperature" contrôle l'aléatoire des réponses du LLM. À 0.1 (très bas), l'IA donne toujours la même réponse la plus probable — parfait pour les JSON structurés. À 0.8 (élevé), l'IA est plus créative et variée — mieux pour la rédaction. Le chat utilise 0.3, l'extraction utilise 0.1.

services/matching.py — L'algorithme de scoring

C'est le cœur intellectuel du projet. L'algorithme calcule un score de compatibilité entre une requête texte et chaque CV en combinant 4 critères pondérés.

🎯 Le scoring hybride multicritères

Le score final est une combinaison pondérée de 4 sous-scores :

🧠 Sémantique
40%
⚡ Compétences
30%
📅 Expérience
20%
📍 Localisation
10%
💡 Formule du score final

score = (sémantique × 0.40) + (compétences × 0.30) + (expérience × 0.20) + (localisation × 0.10) — Le résultat est multiplié par 100 pour obtenir un pourcentage de 0 à 100%.

🧠 Les Embeddings — Le score sémantique

Un "embedding" transforme un texte en une liste de nombres (vecteur). Des textes similaires produisent des vecteurs proches dans l'espace mathématique.

Pythonmatching.py — Initialisation du modèle d'embeddings
from sentence_transformers import SentenceTransformer

_model = SentenceTransformer("paraphrase-multilingual-MiniLM-L12-v2")
# Ce modèle :
# - Est multilingue (français, anglais, etc.)
# - Encode n'importe quel texte en 384 nombres (vecteur)
# - Des textes similaires → vecteurs proches → cosinus élevé

def encoder(texte: str) -> list[float]:
    return _model.encode(texte, normalize_embeddings=True).tolist()
    # normalize=True → le vecteur a une longueur de 1 (simplifie le calcul cosinus)
PythonSimilarité cosinus — Comment comparer deux vecteurs
def cosinus(a: np.ndarray, b: np.ndarray) -> float:
    d = float(np.linalg.norm(a) * np.linalg.norm(b))  # Longueur de chaque vecteur
    return 0.0 if d == 0 else float(np.dot(a, b) / d)
    # np.dot = produit scalaire des deux vecteurs
    # Résultat entre -1 et 1. Plus proche de 1 = plus similaire.
    # Textes identiques → cosinus = 1.0 (100% similaire)
    # Textes sans rapport → cosinus ≈ 0.0
🎓
Exemple concret d'embedding

"Chef de cuisine" → [0.12, -0.45, 0.78, ...] (384 nombres)

"Cuisinier professionnel" → [0.11, -0.44, 0.79, ...] (très similaire !)

"Développeur PHP" → [-0.32, 0.19, -0.12, ...] (très différent)

Le cosinus entre "Chef de cuisine" et "Cuisinier" sera ≈0.95. Entre "Chef de cuisine" et "Développeur PHP" sera ≈0.05. C'est ainsi que l'IA comprend le sens des mots.

⚡ _score_competences()

PythonScoring par correspondance de compétences
_ALIAS = {"js":"javascript", "py":"python", "node":"nodejs"}
# Dictionnaire de synonymes : "js" est traité comme "javascript"

def _match_comp(c1: str, c2: str) -> bool:
    a, b = c1.lower().strip(), c2.lower().strip()
    if a == b or a in b or b in a: return True  # "react" in "reactjs" → True
    return _ALIAS.get(a, a) == _ALIAS.get(b, b)    # "js" → "javascript" == "javascript"

def _score_competences(requete: str, competences: list, texte: str) -> float:
    mots = [m for m in re.findall(r"\w+", requete.lower()) if len(m) > 2]
    # Extrait tous les mots de plus de 2 caractères de la requête
    if not mots: return 0.5  # Si requête vide → score neutre

    txt = (texte or "").lower()
    trouves = sum(1 for m in mots if
                  any(_match_comp(m, c) for c in competences)  # Dans les compétences IA
                  or m in txt)                                    # Ou dans le texte brut du CV
    return trouves / len(mots)  # Ratio : 5 mots trouvés / 10 mots cherchés = 0.5 (50%)

📅 _score_experience()

def _score_experience(requete: str, annees_cv: float) -> float:
    q = requete.lower()
    nums = re.findall(r"(\d+\.?\d*)\s*(?:ans?|an)", q)  # Cherche "3 ans" ou "5 an" dans la requête

    if nums:
        demandees = float(nums[0])   # "3 ans exp." → demandees = 3.0
    elif any(m in q for m in MOTS_SENIOR):  # "senior" → suppose 5 ans
        demandees = 5.0
    else:
        return 0.7  # Pas d'info sur l'expérience → score neutre

    ratio = annees_cv / demandees
    if ratio >= 1.0:
        return min(1.0, 0.70 + (ratio - 1.0) * 0.15)
        # A plus que demandé → bonus progressif, max 1.0 (100%)
    return round(ratio * 0.70, 3)
    # A moins que demandé → score proportionnel. 1 an pour 3 demandés = 0.33 × 0.7 = 23%

📍 _score_localisation()

PythonScoring de localisation — Adapté à l'Afrique de l'Ouest
VILLES_AOF = ["abidjan", "bouaké", "dakar", "bamako", "lomé", ...]
# Liste des villes principales d'Afrique de l'Ouest

def _score_localisation(requete: str, ville: str) -> float:
    if not ville: return 0.3      # Pas de ville → pénalité légère
    q, v = requete.lower(), ville.lower().strip()
    if v in q or q in v: return 1.0  # Ville exacte dans la requête → 100%
    if any(c in q for c in VILLES_AOF) and v in VILLES_AOF:
        return 0.6  # Requête veut une ville AOF, candidat est en AOF → bonus partiel
    if not any(c in q for c in VILLES_AOF + ["paris", "lyon"]):
        return 0.7  # Pas de ville spécifiée dans la requête → score neutre
    return 0.2      # Villes incompatibles → pénalité forte

🏆 classer_candidats() — L'algorithme principal

PythonClassement final des candidats
def classer_candidats(requete: str, candidats: list[dict]) -> list[dict]:
    if not candidats: return []

    # Encoder la requête une seule fois (optimisation)
    vreq = np.array(encoder(requete), dtype=np.float32)

    resultats = []
    for c in candidats:
        texte = c.get("raw_text", "") or ""
        annees = _parse_annees(c.get("annees_exp", 0))
        ville = c.get("ville", "") or c.get("ville_cv", "") or ""

        # Score sémantique : compare le vecteur de la requête avec celui du CV
        vc = depuis_json(c.get("embedding_json"))  # JSON → tableau numpy
        if vc is not None and vc.shape == vreq.shape:
            s_sem = max(0.0, cosinus(vreq, vc))  # Entre 0 et 1
        else:
            s_sem = 0.0  # Pas d'embedding → score sémantique nul

        s_comp = _score_competences(requete, c.get("competences", []), texte)
        s_exp  = _score_experience(requete, annees)
        s_loc  = _score_localisation(requete, ville)

        # Score final pondéré
        score = (s_sem*0.40 + s_comp*0.30 + s_exp*0.20 + s_loc*0.10) * 100

        if score >= 15:  # Seuil minimum : on n'affiche pas les scores < 15%
            resultats.append({
                **c,           # Décompresse tout le dictionnaire candidat (** = spread)
                "score": round(score, 1),
                "_score_detail": {   # Détail pour l'affichage
                    "semantique":   round(s_sem*100, 1),
                    "competences":  round(s_comp*100, 1),
                    "experience":   round(s_exp*100, 1),
                    "localisation": round(s_loc*100, 1),
                }
            })

    resultats.sort(key=lambda x: x["score"], reverse=True)  # Tri décroissant
    return resultats[:5]  # Retourne seulement les 5 meilleurs
💡 L'opérateur ** en Python (spread)

{**c, "score": 87} crée un nouveau dictionnaire qui contient toutes les paires clé:valeur de c, plus la clé "score" avec la valeur 87. C'est l'équivalent de copier le dictionnaire et y ajouter des champs. En JavaScript, l'équivalent est {...c, score: 87}.

Architecture
Les flux complets — De bout en bout

Maintenant que tu connais chaque fichier, voici comment ils s'enchaînent dans les scénarios réels d'utilisation.

🔄 Flux 1 : Inscription et upload de CV (Candidat)

1
Candidat remplit le formulaire d'inscription
login_candidat.php reçoit les données POST. Valide email, longueur du mot de passe, correspondance des mots de passe. Hash le mot de passe avec BCrypt. Insère dans la table candidats. Crée la session. Redirige vers le dashboard.
2
Candidat dépose son CV sur le dashboard
dashboard_candidat.php traite l'action POST upload_cv. Appelle uploadCV() depuis config.php : vérifie l'extension, la taille, renomme le fichier et le déplace dans /uploads/. Insère un enregistrement dans la table cv_files avec le nom du fichier.
3
PHP appelle l'API Python pour indexer
callAPI('/candidats/upload/{id}') utilise cURL pour appeler routes/candidats.py. Python trouve le fichier CV, lit son contenu avec services/cv.py (pdfplumber pour PDF, pytesseract pour images).
4
Python extrait les compétences via Groq IA
services/ia.py envoie les 4000 premiers caractères du texte à Llama 3 via Groq. L'IA retourne un JSON avec les compétences techniques. Ex: ["PHP", "MySQL", "JavaScript", "Laravel"].
5
Python génère l'embedding vectoriel
services/matching.py encode le texte complet du CV avec Sentence-Transformers. Produit un vecteur de 384 nombres flottants. Ce vecteur est converti en JSON et stocké dans cv_files.embedding_json.
6
Tout est stocké en base de données
La table cv_files est mise à jour : texte_brut = texte du CV, embedding_json = vecteur, competences = JSON des compétences. Le candidat est maintenant "indexé" et apparaîtra dans les recherches.

🔍 Flux 2 : Recherche IA (Recruteur)

1
Recruteur tape sa requête dans le dashboard
dashboard_recruteur.php — la fonction JavaScript runSearch() capture le texte. Le navigateur envoie directement une requête POST à http://localhost:8000/recherche avec le texte et la clé API.
2
Python valide le métier via IA
routes/recherche.py appelle valider_requete_metier(). Groq Llama 3 vérifie si le métier existe, corrige les fautes ("developpeur" → "développeur"), liste les compétences attendues. Si invalide, retourne une erreur.
3
Sync des profils externes
services/cv.py → sync_profils_externes() parcourt le dossier /uploads/ pour trouver des CV qui n'appartiennent à aucun compte (déposés manuellement). Les indexe en BDD si nouveaux ou modifiés.
4
Chargement de tous les CV depuis la BDD
Deux requêtes SQL : une pour les candidats inscrits (avec leur dernier CV), une pour les profils externes. Résultats fusionnés en une seule liste. Les compétences JSON sont désérialisées en listes Python.
5
Algorithme de matching — scoring multicritères
services/matching.py → classer_candidats(). Pour chaque candidat : calcul du cosinus entre la requête et le vecteur CV (sémantique), comptage des compétences communes, comparaison des années d'expérience, vérification de la ville. Score pondéré 40/30/20/10.
6
Génération des résumés IA
Pour chacun des top 5 candidats, generer_resume_candidat() appelle Groq pour écrire 2-3 phrases expliquant pourquoi ce profil correspond ou non. Commence par "Profil pertinent", "Profil partiel" ou "Profil non adapté".
7
Retour et affichage
Le JSON des 5 profils avec leurs scores est retourné au navigateur. renderResults() génère dynamiquement les cartes HTML avec le rang, les scores détaillés, les compétences et le résumé IA. L'agent de chat est prêt.

💬 Flux 3 : Conversation avec l'agent IA

1
Recruteur envoie un message au chat
sendChat() récupère le message. Ajoute à chatHistory (tableau JavaScript). Affiche une bulle "typing" animée pendant l'attente. Envoie POST à /conversation/chat avec le message, l'historique complet et les profils actuels.
2
Python enrichit le contexte
_charger_textes_cv() charge le texte brut de chaque CV depuis la BDD (max 3000 caractères par CV). Construit un résumé structuré de chaque profil pour le prompt.
3
Construction du prompt complet
Messages = [système avec contexte CV] + [historique] + [nouveau message]. Le tout est envoyé à Groq Llama 3.3-70B (modèle plus puissant que pour la recherche).
4
Réponse et sauvegarde
La réponse de l'IA est nettoyée (suppression des blocs ```json``` parasites). Message utilisateur + réponse IA sauvegardés dans conversation_history en BDD. Retournés au navigateur pour affichage.

✉️ Flux 4 : Contact candidat (messagerie)

1
Recruteur clique "Contacter"
contactCandidat(userId, nomCandidat) ouvre le modal HTML. Remplit le champ caché avec l'ID du candidat. Affiche son nom dans le sous-titre du modal.
2
Recruteur remplit et envoie le message
envoyerMessage() lit le sujet et le corps. Envoie un POST (JSON) à envoyer_message.php. Ce fichier vérifie que l'utilisateur est bien un recruteur, que le destinataire existe, puis insère dans la table messages.
3
Le candidat reçoit le message
Sur son dashboard, le badge "Messages" affiche le nombre de messages non-lus. En cliquant sur "Messages", loadMessages() appelle get_messages.php qui retourne les messages en JSON. Affichage avec bordure colorée si non-lu.

🔐 Sécurité

🛡️
Injection SQL

Attaque où un utilisateur insère du SQL dans un formulaire. Protection : requêtes préparées PDO avec ? comme paramètre. Le texte utilisateur ne devient jamais du SQL.

🛡️
XSS (Cross-Site Scripting)

Attaque où du JavaScript malveillant est injecté dans la page. Protection : htmlspecialchars() en PHP convertit <script> en texte inoffensif.

🛡️
Path Traversal

Attaque où ../../../etc/passwd est utilisé pour accéder à des fichiers interdits. Protection dans cv_view.php : suppression des .. et vérification que le chemin est dans /uploads/.

🛡️
Authentification API

L'API Python vérifie le header X-API-Key à chaque requête. Si la clé ne correspond pas à API_KEY, la requête est rejetée avec HTTP 401.
